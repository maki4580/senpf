# 用語集(データ連携仕様書 補助資料)

- 対象文書: [data-integration-spec_new.md](data-integration-spec_new.md)
- 目的: 仕様書に出てくる専門用語・略語を、**予備知識のない方でも読めるように**平易に説明する
- 読み方: 各表は「用語 / やさしい説明 / この仕様書での使われ方」の3列。まず左と真ん中だけ読めば全体像がつかめます

> この基盤がやろうとしていることを一言でいうと——
> **たくさんの拠点(お客さまの施設)に置かれた防犯カメラの映像をAIが解析し、「人が侵入した」などの検知イベントを本社のシステムに集め、後から映像と突き合わせて確認したり、AIの精度を改善したりする仕組み**です。以下の用語はこの流れの中で登場します。

## 0. この仕様書だけで使われる記号・ID

文書独自のラベルです。意味を知らないと本文が暗号に見えるので最初に説明します。

| 記号 | やさしい説明 | 補足 |
|---|---|---|
| **IF** | インターフェース(Interface)の略。「システムAとシステムBの間で、どんなデータを・どんな方法でやりとりするか」という**つなぎ目**のこと | この仕様書の主役。全部で40本以上ある |
| **IF-S / W / H / D / M / P / F** | IFを場所・方向で分類した記号。**S**=現場の中、**W**=現場→本社、**H**=本社の中、**D**=本社→現場(下り)、**M**=保守作業・クラウド連携、**P**=初期設定・証明書、**F**=運用管理 | 例: `IF-W01` は「現場→本社の1番目のつなぎ目」 |
| **D1〜D17** | 別紙([design-decisions.md](design-decisions.md))に記録された**設計上の決定事項**の番号 | 例: D9=「通信は必ず現場側から始める」という決定 |
| **N1〜N9** | **まだ決まっていない未確定事項**の番号(要検討リスト) | 例: N6=性能が足りるかの検証待ち |
| **A5〜A11** | 別紙に元からある**宿題(未検討課題)**の番号 | 本仕様書はこれを引き継いでいる |
| **◎ / ○ / △** | IFの確度。◎=確定、○=本書で仕様化(要レビュー)、△=未確定(条件付き) | 一覧表の「状態」列で使用 |
| **★** | 「図に描かれていないが、次回の図の更新で追加を推奨」という印 | — |

## 1. 登場する製品・ソフトウェア

具体的な製品名やソフトの名前です。

| 用語 | やさしい説明 | この仕様書での役割 |
|---|---|---|
| **IoT** | Internet of Things。カメラや機械などの「モノ」をネットにつなげてデータを集める仕組みの総称 | この基盤全体のジャンル |
| **Cumulocity(クムロシティ)** | ドイツ製の**IoTプラットフォーム製品**。多数の機器を登録・監視し、機器から届くデータを貯めて画面で見られるようにする土台 | この基盤の中心。機器管理の本体 |
| **Cumulocity Edge(エッジ版)** | Cumulocityを**自社の建物内のサーバーに設置して使う版**(クラウドではなくオンプレミス) | 本社サーバーに置いて全拠点を集約 |
| **thin-edge.io(シンエッジ)** | 現場の機器に入れる**小さな中継ソフト**。現場データをCumulocityが分かる形に変換して送る | 現場と本社をつなぐ通訳役 |
| **Genetec(ジェネテック)** | カナダ製の**映像監視システム(VMS)製品**。カメラ映像の録画・再生・配信を担う | 映像の管理はこの製品に任せる |
| **Genetec Security Center / GSC** | Genetec製品の中核。複数拠点の映像を**中央で統合管理**するソフト | 本社で映像を束ねる |
| **Keycloak(キークローク)** | **ログイン認証を一手に引き受けるソフト**(IdP)。一度ログインすれば複数アプリを使い回せる(SSO) | 全システム共通のログイン窓口 |
| **Grafana Alloy(グラファナ・アロイ)** | 各機器の**動作ログや稼働状況を集めて転送する**ソフト | 現場の監視データを本社へ送る |
| **OpenTelemetry / Otel / OTLP** | システムの稼働状況(速度・エラー・ログ)を**メーカー横断の共通形式で集める**仕組み。OTLPはその通信規格 | 監視データの共通言語 |
| **Docker Compose(ドッカー・コンポーズ)** | 複数のソフトを**コンテナという小箱に入れてまとめて動かす**仕組み | 現場の解析装置の構成方式 |
| **MongoDB(モンゴDB)** | 柔軟な形式でデータを貯められる**データベース**の一種 | Cumulocityが内部で使う保管庫 |
| **ベクトルDB** | 「特徴量」(下記)のような**数値の並びを貯めて、似たものを高速に探せる**データベース | AI学習用データの保管庫 |
| **オブジェクトストレージ** | 写真・動画・ファイルを**大量に安く長期保存できる**保管庫 | 長期保存はここ。S3という規格に対応 |
| **S3 / S3互換API** | Amazonが作ったオブジェクトストレージの**事実上の標準規格**。これに合わせた製品を「S3互換」と呼ぶ | 長期保管庫の操作方法 |
| **Parquet(パーケット)** | 大量データを**分析しやすく圧縮して保存するファイル形式** | 長期保存データの形式 |
| **SQLite(エスキューライト)** | 1ファイルで動く**軽量なデータベース** | 重複チェック用の小さな記録に使用 |
| **mosquitto(モスキート)** | MQTT(下記)通信を仲介する**メッセージ配達係**のソフト | 現場でデータを一時的に溜める役 |
| **DataHub / DataHub Edge** | Cumulocity公式の、貯めたデータを分析用保管庫に**書き出すオプション機能** | 自作の書き出し処理と比較検討中(N7) |
| **Dremio(ドレミオ)** | DataHubが内部で使う**データ分析エンジン** | 導入するなら追加コスト要因 |
| **Apama / EPL / Streaming Analytics** | Cumulocity上で**流れてくるデータにリアルタイムで反応するルールを書く**仕組み。EPLはその専用言語 | 通知ルールの実装候補(要ライセンス確認) |
| **Smart Rules(スマートルール)** | Cumulocity標準の、**プログラミング不要で作れる簡単な自動ルール** | 「〜なら通知」等。ただし機能に限界あり |
| **Notification 2.0** | Cumulocityの、**イベント発生を外部に知らせる購読の仕組み** | 通知機能の実装候補の一つ |
| **sm-plugin** | thin-edgeの拡張部品。ここでは**AIモデルの入れ替えを実行する自作部品** | モデル更新の現場側の実行役 |
| **Kubernetes / k3s** | 多数のコンテナを**自動で管理する仕組み**。k3sはその軽量版 | Cumulocity Edgeの設置形態の一つ |
| **OVA** | 仮想マシンを**丸ごと1ファイルに固めた配布形式**(アプライアンス) | Cumulocity Edgeの設置形態の一つ |
| **Ansible(アンシブル)** | 多数のサーバーの**設定作業を自動化する**ツール | 保守での構成管理に使用 |
| **VMS / NVR** | VMS=映像監視システム、NVR=ネットワーク録画装置。どちらも**カメラ映像を録画・管理する**機器/ソフト | 現場のGenetecがこれに当たる |

## 2. 通信のプロトコル・規格

「どういう約束事でデータを送るか」の規格名です。

| 用語 | やさしい説明 | この仕様書での使われ方 |
|---|---|---|
| **プロトコル** | 機器同士が会話するための**共通の手順・お作法**のこと | 各IFで「どの手順を使うか」を指定 |
| **RTSP** | **映像を流すための規格**。ライブ配信や録画映像の受け渡しに使う | カメラ→VMS→解析への映像 |
| **ONVIF(オンヴィフ)** | 監視カメラの**メーカー間共通規格**。機種が違っても同じ手順で操作・確認できる | カメラの死活(生きているか)確認 |
| **MQTT(エムキューティーティー)** | **少ないデータ量で効率よくメッセージを送る**IoT向け規格。回線が細くても強い | 検知イベントの送信の主役 |
| **HTTP / HTTPS** | Webでおなじみの**データ送受信の規格**。HTTPSはそれを暗号化して安全にしたもの | 画面操作・画像添付・API連携 |
| **REST(レスト)** | HTTPを使って**データを取得・登録する標準的な作法**(API設計の流儀) | 各種データの読み書き |
| **API** | Application Programming Interface。**ソフト同士がデータをやりとりするための窓口**。人間のUIに対し、プログラム用の入口 | 「REST API」などの形で頻出 |
| **APIトークン** | APIを使うときの**合言葉(鍵)**。人ではなく機械が使う認証情報 | 機器・システム間の認証 |
| **TLS / mTLS** | 通信を**暗号化して盗み見を防ぐ**技術。mTLSは送り手と受け手が**お互いに身元証明**する強化版 | 全ての外部通信の暗号化 |
| **gRPC(ジーアールピーシー)** | 高速な**データ通信の規格**の一つ | 監視データ転送(OTLP)で使用 |
| **ICMP** | ネットワークで**「そこにいますか?」と確認する**基本的な仕組み(pingの中身) | カメラの生存確認 |
| **NTP** | 全機器の**時計を正確に合わせる**仕組み | 「いつ起きたか」を正しく記録するため必須 |
| **SMTP** | **メールを送る**ための規格 | 通知メールの送信 |
| **Webhook(ウェブフック)** | 何かが起きたとき、**指定したURLに自動でお知らせを飛ばす**仕組み | 案件アプリへの通知 |
| **OIDC(オーアイディーシー)** | **安全にログインを実現する標準規格**(OpenID Connect)。Keycloakが使う | 利用者のログイン全般 |
| **SSO** | Single Sign-On。**一度ログインすれば複数のアプリを使い回せる**仕組み | 全アプリ共通のログイン体験 |
| **Authorization Code Flow** | OIDCの**標準的なログイン手順**の一つ(安全性が高い方式) | ログインの具体的な流れ |
| **X.509 / デバイス証明書** | 機器の**身分証明書(電子的な鍵)**。本物の機器だけが通信できるようにする | 現場機器の認証 |
| **SDK** | Software Development Kit。**その製品と連携するプログラムを作るための道具箱** | Genetec連携で使用 |
| **Federation(フェデレーション)** | 複数の映像システムを**連携・統合する**Genetecの機能 | 現場VMSと本社GSCの連携 |
| **UUID / UUIDv7** | 世界で**重複しない一意のID**。v7は時刻順に並ぶ新しい版 | イベントの識別子(eventUuid) |
| **RFC 9562** | UUIDの**正式な仕様書(国際的な取り決め)**の番号 | UUIDv7の表記の根拠 |
| **ISO 8601** | 日時の**国際的な書き方の標準**(例: 2026-07-20T10:23:45+09:00) | タイムスタンプの形式 |

## 3. システム構成・ネットワークの用語

「どこに何を置くか」「どう区切るか」の話です。

| 用語 | やさしい説明 | この仕様書での使われ方 |
|---|---|---|
| **エッジ / エッジ端末** | 本社ではなく**現場(データが生まれる場所)に置く機器**。その場で処理する | 現場の「画像解析装置」がこれ |
| **ゲートウェイ** | 異なるネットワークやシステムの**出入口・橋渡し役**の機器/ソフト | thin-edgeやBOXアダプタ |
| **VM1 / VM2** | Virtual Machine(仮想マシン)。1台の物理サーバー内に作る**仮想的なコンピュータ**。VM1=基盤の土台、VM2=アプリ層 | 本社サーバーの論理的な区画 |
| **オンプレミス(オンプレ)** | クラウドを使わず、**自社の建物内にサーバーを置いて運用**すること | 本社サーバールームの方式 |
| **クラウド** | インターネット越しに使う、**他社が運用する共有サーバー** | AIモデル改善サービスの置き場所 |
| **LAN / WAN** | LAN=**建物内の狭いネットワーク**、WAN=**拠点間を結ぶ広いネットワーク** | 現場内=LAN、現場⇔本社=WAN |
| **閉域網(へいいきもう)** | **インターネットを経由しない、専用の閉じたネットワーク**。外部から入れず安全 | 現場と本社を結ぶ回線 |
| **VPN** | 公衆回線上に作る**暗号化された専用トンネル**。安全に遠隔接続できる | 保守作業員が本社につなぐ経路 |
| **FW(ファイアウォール)** | 許可した通信だけ通し、**不正な通信を遮断する防火壁** | 経路の制限に使用 |
| **ACL** | Access Control List。**「誰・どこからの通信を許すか」の許可リスト** | 旧機器の代替的な守り |
| **セグメント** | ネットワークを**用途ごとに区切った区画**。区切ることで被害を局所化 | 機器の種類ごとに分離 |
| **FQDN** | インターネット上の**完全なドメイン名**(例: example.com)。接続先を厳密に指定 | クラウド接続先を絞る条件 |
| **アウトバウンド / 限定アウトバウンド** | 内側から外側(クラウド)への通信。**限定**=特定の宛先だけ許可 | 本社→クラウドの厳格な制御 |
| **プロビジョニング** | 新しい機器・拠点を**使えるように初期設定・登録する**作業 | 拠点追加時の一連の手順 |
| **enrollment(エンロールメント)** | 機器を**最初にシステムに登録する**こと(証明書の発行など) | 証明書のライフサイクルの入口 |
| **CA** | Certificate Authority。**証明書を発行する認証局** | デバイス証明書の発行元 |
| **失効(しっこう)** | 発行済みの証明書・権限を**無効にする**こと(盗難・撤去時など) | 証明書ライフサイクルの一部 |

## 4. データの種類に関する用語

やりとりされる「中身」の呼び名です。

| 用語 | やさしい説明 | この仕様書での使われ方 |
|---|---|---|
| **イベント(Event)** | 「人が侵入した」など、**何かが起きたという記録**。Cumulocityのデータ種別の一つ | 検知の記録(x_Detection等) |
| **アラーム(Alarm)** | イベントの中でも**対応が必要な通報レベルのもの**。状態(発生中/解除)を持つ | カメラ故障・侵入通報など |
| **計測(Measurement)** | 温度や応答速度など、**数値の測定データ** | カメラの死活監視データ |
| **ペイロード** | 通信で運ばれる**データの中身(荷物)本体** | イベントの実データ部分 |
| **スナップショット** | イベント発生の瞬間に撮った**静止画(1枚の写真)** | 検知時の証拠画像 |
| **クリップ** | イベント前後を**切り出した短い動画** | 後から確認する映像 |
| **特徴量(ベクトル)** | 画像から抽出した、**AIが扱う数値の並び**。人の顔や物体の特徴を数値化したもの | AIモデル改善の材料 |
| **dims(次元数)** | 特徴量の数値がいくつ並んでいるかの**個数** | 例: 512個の数値なら512次元 |
| **生体情報 / 個人識別符号** | 顔データなど、**個人を特定できてしまう情報**。法律上、特別に慎重な扱いが必要 | 持ち出しを厳格に制限 |
| **仮名化(かめいか)** | 誰の情報か**すぐには分からないように加工**すること | クラウド送信前の保護策 |
| **メタデータ** | データそのものではなく、**それに付随する情報**(いつ・どのカメラ・どんな種類か等) | イベントの属性情報 |
| **サイドカーJSON** | 画像・動画ファイルに**寄り添って置かれる説明書ファイル**(撮影時刻やカメラID等を記載) | 保存映像の付属情報 |
| **テレメトリ** | 機器やソフトの**稼働状況を遠隔で集めたデータ**(速度・エラー・ログ) | システム自体の健康診断データ |
| **メトリクス / ログ / トレース** | メトリクス=数値の測定、ログ=動作の記録、トレース=処理の流れの追跡。**監視の3本柱** | Otelで集める対象 |
| **modelVersion(モデル版数)** | 使っているAIモデルの**バージョン番号**。どの版で検知したか記録する | 検知データに必ず付ける |
| **モデル / AIモデル** | 画像から異常を見つける**AIの頭脳部分**(学習済みの判断ロジック) | 定期的に更新して精度改善 |
| **推論エンジン** | AIモデルを実際に動かして**判断を実行する部分** | 現場で映像を解析 |
| **カスタムフラグメント** | Cumulocityのデータに**独自の追加項目を付ける仕組み**。`x_`で始める約束 | x_Detection等の独自データ |
| **Managed Object / Inventory** | Cumulocityが管理する**機器や資産の登録情報(台帳)**。Inventoryはその台帳全体 | カメラ・機器の登録先 |
| **child device(子デバイス)** | ゲートウェイの**配下にぶら下がる機器**(カメラやBOX)。親を通じて管理される | 各カメラの管理単位 |
| **Operational Store** | Cumulocityが**最近のデータを貯めておく作業用の保管庫**(目安90日で消える短期保管) | 短期データの置き場所 |

## 5. データの信頼性・エラー処理の用語

「通信が途切れてもデータを失わない」ための工夫の呼び名です。この仕様書の重要テーマです。

| 用語 | やさしい説明 | この仕様書での使われ方 |
|---|---|---|
| **バッファ / スプール / キュー** | データを**一時的に溜めておく待合室**。送れないとき溜めておき、後で送る | 回線が切れた時の待避先 |
| **ストア&フォワード** | **いったん保存(ストア)して、送れるようになったら転送(フォワード)**する仕組み | thin-edgeの網切れ対策 |
| **WAL** | Write-Ahead Log。**先に記録してから処理する**ことでデータ消失を防ぐ手法 | 監視データの一時保存 |
| **at-least-once(アットリーストワンス)** | 「**最低1回は必ず届ける**」方式。ただし再送で**同じデータが2回届くことがある** | 再送方式の性質(だから重複対策が要る) |
| **冪等性(べきとうせい) / 冪等化** | **同じ操作を何回繰り返しても結果が変わらない**性質。2回届いても1回分として扱える | 重複しても壊れない設計 |
| **重複排除(de-duplicate)** | **同じデータの2回目以降を捨てる**処理 | 再送でダブったイベントの始末 |
| **eventUuid** | 各イベントに付ける**世界で唯一のID**。これを鍵に「同じイベントか」を判定 | 重複排除・追跡の共通キー |
| **upsert(アップサート)** | **あれば更新、なければ新規追加**する書き込み方(update + insert) | 特徴量の重複しない登録 |
| **デッドレター** | **処理できなかったデータを捨てずに退避させておく箱**。後で調べられる | 変換不能データの避難先 |
| **リトライ / 指数バックオフ** | 失敗時に**再試行**すること。指数バックオフ=**待ち時間を徐々に延ばしながら**再試行 | 送信失敗時の再挑戦 |
| **ヒステリシス** | 判定が**頻繁にオン・オフを繰り返さないよう、余裕(あそび)を持たせる**こと | カメラ故障判定の安定化 |
| **フラッピング** | 状態が**短時間でパタパタ切り替わる**現象(故障→復旧→故障…) | ヒステリシスで防ぐ対象 |
| **保持ウィンドウ** | 重複チェックのために**過去どこまでさかのぼって記憶しておくか**の期間 | 例: 72時間 |
| **リプレイ(再送の副作用)** | 回線復旧時に**古いデータがどっと流れ込み、今起きたかのように誤って通知される**問題 | 「古すぎる通知は抑える」ガードで対策 |
| **オフロード** | 短期保管庫から長期保管庫へ**データを移し替える**こと | 古いデータの長期保存 |
| **ルックバック** | 取りこぼし防止に、**少し前の期間を毎回さかのぼって取り直す**こと | 遅れて届いたデータの回収 |

## 6. セキュリティ・運用・性能の用語

安全性・止まらなさ・処理能力に関する言葉です。

| 用語 | やさしい説明 | この仕様書での使われ方 |
|---|---|---|
| **IdP** | Identity Provider。**ログイン(本人確認)を専門に担う仕組み** | Keycloakがこの役割 |
| **ロール** | 利用者に与える**役割・権限のまとまり**(例: 閲覧のみ、管理者) | 「読み取り専用ロール」等 |
| **最小権限** | 各人・各システムに**必要最小限の権限しか与えない**セキュリティ原則 | 権限設計の基本方針 |
| **監査ログ** | 「**誰が・いつ・何をしたか(見たかも含む)**」を後から検証できるよう残す記録 | 顔画像の閲覧なども記録 |
| **マルチテナント / テナント** | 1つのシステムを**複数の顧客で区切って共有**する方式。テナント=1顧客分の区画 | 今回は1顧客なので不要と判断(D17) |
| **資産階層 / グループ** | 機器を**拠点ごとなど階層的にグループ分け**して管理する仕組み | 拠点単位の区分に使用 |
| **スループット / tps** | 一定時間に**処理できるデータ量**。tps=1秒あたりの件数(transactions per second) | 性能が足りるかの試算 |
| **垂直スケール / 水平スケール** | 垂直=**1台を高性能にする**、水平=**台数を増やす**。Cumulocity Edgeは垂直のみ | 性能の増やし方の制約 |
| **クラスタ** | **複数台をまとめて1つのように動かす**構成(水平スケールや冗長化に使う) | Edgeは非対応という制約 |
| **HA(高可用性)** | High Availability。**故障してもサービスが止まりにくい**構成 | 止まらない工夫の要否 |
| **SPOF** | Single Point Of Failure。**そこが壊れると全体が止まる急所** | 中央集約による弱点 |
| **PoC** | Proof of Concept。**本番前に「本当に実現できるか」を試す検証** | 契約前の性能確認 |
| **RPO / RTO** | RPO=**どこまでのデータ消失を許すか**、RTO=**復旧までにかける時間の目標** | バックアップ設計の目標値 |
| **SLO** | Service Level Objective。**サービス品質の目標水準** | 許容する停止時間などの基準 |
| **バックアップ / リストア** | バックアップ=**データの控えを取る**、リストア=**控えから復元する** | 障害復旧の備え |
| **ロールアウト / 段階配布** | 更新を**一度に全部でなく、少しずつ様子を見ながら展開**すること | AIモデル更新の配り方 |
| **bulk operation(バルク操作)** | 多数の機器に**まとめて同じ操作**を指示すること | モデルの一括配布 |
| **ロールバック** | 更新して問題が出たとき**元の状態に戻す**こと | モデル適用失敗時の復旧 |
| **Operation(オペレーション)** | Cumulocityが機器に送る**遠隔操作の指示**(例: ソフト更新せよ) | モデル配布の指示 |
| **サプライチェーン(対策)** | 部品や更新物の**流通経路のどこかが汚染される攻撃**への備え | モデルの署名・ハッシュ検証 |
| **ハッシュ / 署名** | ファイルが**改ざんされていないか・本物かを確認する電子的な照合値** | 配布モデルの正当性検証 |
