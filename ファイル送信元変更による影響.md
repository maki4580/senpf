## 前提の整理 — 争点は「RESTの出口をどこにするか」だけ

まず認識合わせです。現行設計([data-integration-spec_new.md](data-integration-spec_new.md) §3.2/§3.5)では、スナップショット付きイベントは**すでにMQTT経由ではなくREST**です(MQTTではmapperが採番するイベントIDを発行元が知り得ないため)。つまり:

- **現状**: パイプライン → thin-edgeの**Cumulocityプロキシ**(localhost:8001 `/c8y/...`)→ Cumulocity Edge
- **提案**: パイプライン → **直接** Cumulocity Edge のREST API

データパスはどちらもHTTP(REST)で、閉域網を渡る物理経路も同一(同じ装置から出る)。違いは**localhost内のワンホップと、認証情報を誰が持つか**だけです。

## 比較

| 観点 | 現状(thin-edgeプロキシ経由) | 提案(直接送信) |
|---|---|---|
| 認証情報 | thin-edgeのX.509デバイス証明書1系統に集約。プロキシがJWT取得・更新を自動処理し、パイプラインは認証情報を一切持たない | パイプライン用の認証情報(拠点別サービスユーザー or APIトークン)を**N拠点分、新規に発行・ローテーション・失効管理**する必要。IF-P01/P02の対象拡大 |
| 権限スコープ | デバイスユーザー権限=自デバイス+child deviceに閉じる。漏洩時の影響範囲が最小 | トークンのスコープ設計を誤ると他拠点のデータに書ける。拠点別に最小権限ユーザーを作り込む運用負荷 |
| 網断時の挙動 | HTTP経路はS&F対象外 → ローカルスプール必須(§3.2) | **同じ**。thin-edgeを外してもRESTのS&Fは存在しないため、スプール実装は消えない |
| 性能 | localhost内ホップが1つ増えるが、ボトルネックは閉域網WANでありプロキシのオーバーヘッドは実質無視できる。既知の問題として**JWT起因の散発的401**がドキュメントに記載あり(リトライで回避) | ホップ減の理論上の利点はあるが体感差はほぼない。TLS終端・証明書検証を自前実装 |
| 障害切り分け・監視 | Cumulocity向け通信の出口がthin-edgeに一元化され、c8y_Availability監視・FW/ACL・監査の観測点が1つ | 出口が2系統になり、切り分け・監視対象が増える |
| thin-edge障害時 | プロキシ停止=添付送信も止まる(ただしMQTT経路も同時に止まっており、スプールが保全する) | thin-edge停止中もREST送信は継続可能 ← **提案の実質唯一の利点** |
| 実装量 | 追加なし(現仕様のまま) | 認証管理・TLS設定・仕様書改訂(§3.2/§3.5/§4.4/IF-P01/P02) |

## 私の見解: **現状(thin-edgeプロキシ経由)を維持すべき**

理由は3点です。

1. **提案の主な動機と思われる「thin-edge依存の排除」の効果が薄い**。直接送信にしても網断対策のスプールは必要なままで、thin-edgeが落ちる障害ではMQTT経路(イベント本体)も止まるため、添付だけ送れても業務動線は成立しません。得られるのは「thin-edgeのみ落ちて閉域網とCumulocityは健在」という狭い障害モードへの耐性だけです。

2. **失うものが大きい**。認証情報がデバイス証明書1系統からN拠点分のトークン管理に増えるのは、証明書ライフサイクル(IF-P01)・構成コード管理(D10)・最小権限の原則に逆行します。50拠点規模ならトークンの発行・ローテーション・失効だけで恒常的な運用負荷になります。

3. **プロキシは2026年7月時点でthin-edge 1.x系の標準機能として安定提供**されており、全REST API・全HTTPメソッド透過、JWT自動処理、child deviceからの利用まで公式サポートされています。「プロキシを介すと機能が制限される」という類の制約はありません。

**ただし条件付きです**。PoC(A7/A8の実測項目)に次を加えることを推奨します:

- プロキシ経由での添付アップロードのスループット・大サイズ時のタイムアウト実測
- 既知の**散発的401**に対するリトライ実装(スプール再送ワーカーに401リトライを組み込む — これは仕様書§3.2に一行追記する価値があります)
- パイプラインの送信部をインターフェースとして抽象化しておく(万一プロキシに実測上の問題が出た場合、直接送信への切り替えコストを小さくしておく)

つまり「設計としては現状維持、ただし切り替え可能な作りにしてPoCで裏を取る」が回答です。他開発者への説明には上の比較表がそのまま使えます。ご希望なら、この判断を設計判断記録(D18)や仕様書§3.2への追記として文書化します。

**これを受けるCumulocity Edge側では、上記2つの方法での送信で違いはある？**

結論から言うと、**Edge側の受信処理・格納・制約はどちらでも完全に同一**です。違いが出るのは「**どのユーザーとしてリクエストが来るか**」とそれに付随する部分だけです。

## 同一な部分

thin-edgeのプロキシはトンネルではなく、単なるローカルHTTPフォワーダーです。どちらの方式でも、Edgeから見えるのは「現場装置のIPから来る、`POST /event/events` と `POST /event/events/{id}/binaries` への通常のHTTPSリクエスト」であり:

- 受信エンドポイント・処理パス・MongoDB(Operational Store)への格納形式は同一
- 添付サイズ上限(テナントオプション/リバースプロキシの`client_max_body_size`)、1イベント1バイナリ制約も同一
- FW/ネットワーク要件も同一(現場→Edge:443の現場発接続。D9整合はどちらも同じ)

つまりEdge側に「受け方の作り分け」は発生しません。

## 違いが出る部分 — 認証主体とその波及

| 観点 | プロキシ経由(現状) | 直接送信(提案) |
|---|---|---|
| Edgeが見る認証主体 | **デバイスユーザー**(thin-edge登録時に自動作成される`device_xxx`。JWTで認証) | 別途用意する**サービスユーザー/APIトークン** |
| Edge側の事前設定 | **追加作業なし**(デバイス登録済みなら動く) | 拠点別ユーザー作成+権限付与(グループ+Inventoryロールで自拠点に絞る設計・運用が必要) |
| 権限の効き方 | デバイスユーザーは製品仕様として**自デバイス+child deviceにしか書けない**。設定ミスの余地がない | Inventoryロールの割当を誤ると他拠点に書ける。「絞る作り込み」を人が正しく維持する前提 |
| 監査ログ・イベントのowner | 作成者=デバイスユーザーなので「どの装置が上げたか」がEdge標準の監査・owner情報で自明 | 作成者=サービスユーザー。装置との対応付けは命名規則等の運用で担保 |
| デバイス可用性(c8y_Availability)への寄与 | デバイスユーザーの通信としてカウントされる | サービスユーザーの通信は装置の死活にカウントされない(実害は小 — 可用性は本線のMQTT接続で維持されるため) |

## 要点

Edge側の観点でも、プロキシ経由の優位は前回の装置側の議論と同じ構図になります。**「デバイスは自分とその配下にしか書けない」という製品組み込みの権限モデルをそのまま使えるか、それを拠点別サービスユーザー+Inventoryロールで自前再現するか**の差です。後者はEdge側にも拠点数分のユーザー管理・ロール割当という恒常的な設定資産が増え、D17の「グループ+Inventoryロールで拠点区分」の設計に、人為ミスで越境しうる書き込み権限の管理という新しい心配事を足すことになります。

受信性能の差はありません。強いて挙げるなら、どちらの方式でもスプール再送ワーカーが復旧後に一斉送信する時のREST同時リクエスト数がEdgeの負荷試験(N6のPoC)に含まれるべき、という点は共通の注意事項です。
