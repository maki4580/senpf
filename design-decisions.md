# IoT基盤 設計判断記録(Design Decision Log)

- 作成日: 2026-07-06
- 作成方法: 構想レビューインタビュー(/grill-me)の結果を整理
- ステータス: 初期構想フェーズ。初号案件合意前
- 更新履歴: 2026-07-06 構成・データフロー設計([architecture-camera-monitoring.md](architecture-camera-monitoring.md))に伴いD2改訂・D13〜D16追加。同設計はCumulocity採用を前提とした(転換2のTCO比較は基準としては残るが、設計はCumulocity Edge前提で先行)

## 0. 構想の出発点

> CumulocityをベースにしてIoT基盤を構築したい。この基盤を基に、他チームのIoTを使用したプロジェクトの早期立ち上げやコストダウンを目指したい。また初期はこの基盤はオンプレとしたい。

インタビューの結果、この出発点は以下の3点で大きく転換された。

---

## 1. 当初構想からの3つの大きな転換

### 転換1: 「自社DCに中央基盤」→「顧客閉域網内に配布するアプライアンス群」

- 初号案件は顧客ネットワーク内に完全に閉じる(外部への業務データ送信なし。保守VPNのみ許可される見込み)
- 顧客の拠点間はネットワークで繋がらないため、**拠点ごとに基盤インスタンスを設置**する
- 規模見通し: 初号案件で数拠点〜10、2年後の全案件合計で10〜50インスタンス
- 自社DCの中央共通基盤は、閉域要件のない将来案件向けの**フェーズ2**に格下げ

### 転換2: 「Cumulocity本命」→「実トポロジー制約下の3年TCO勝負」

- 制約条件: 拠点分散・閉域・多インスタンス(10〜50)・運用体制が薄い
- インスタンス単位ライセンスの商用製品はこのトポロジーで経済性が厳しい
- **判定方法: Cumulocity Edge vs OSSベース(ThingsBoard CE/PE等)で3年TCOを見積もり比較する**
- Cumulocityベンダーには実トポロジー(拠点分散・多インスタンス・閉域)を開示して見積りを取る
- 予想: この条件ではOSSベースが優位になる公算が高い

### 転換3: 「IoT製品の導入」→「フリート管理こそが中核プロダクト」

インタビューで判明した最大の盲点。閉域網内に散在する10〜50インスタンスを薄い体制で維持するには、以下3点が基盤チームの本当の中核成果物になる。IoT製品自体は配布物の中身の一部に過ぎない。

1. **標準サイトサーバー像**: ハードウェア推奨仕様+OS+IoT基盤+エージェント一式を単一イメージ/構成コードで定義。全拠点を同一構成に強制する
2. **構成管理の一元化**: Ansible等で「全拠点に同じ変更を流せる」状態を1拠点目から作る(後付けは不可能)
3. **メタ監視**: 各インスタンスが保守VPN経由で自社側に死活・バージョン情報を報告する仕組み

## 2. 確定した設計判断

| # | 判断 | 内容 | 根拠 |
|---|------|------|------|
| D1 | 映像は基盤の責務外 | 映像ストリーム(録画・ライブ視聴・保存・配信)はVMS/NVR/現地エッジの領域。基盤は機器管理+イベント+通知に徹する | 帯域・ストレージコストがコスト目標を破壊する。IoT基盤製品の設計対象外 |
| D2 | 顧客向け画面は基盤の責務外 **(2026-07-06改訂: 例外1件)** | 顧客向けUIは案件側アプリがAPI経由で構築。基盤の組み込みダッシュボードは「あるがまま」提供(カスタマイズ請負なし)。**改訂: イベント/アラームと映像を紐づけて確認するアプリに限り基盤標準部品に格上げ。**提供範囲はアラーム一覧・絞り込み・該当映像再生・クリップ保存指示まで。業務フロー・レイアウトカスタム・VMS固有機能の露出は引き続き案件側 | UIカスタマイズ要求は無限に湧き、基盤チームが案件の下請け化する。改訂理由: 「検知→映像確認」は監視カメラ全案件共通の中核動線であり、案件ごとの再実装はコストダウン目的に反する。カスタマイズ境界を明文化して下請け化リスクを抑える |
| D3 | 初期スコープは最小3点 | 死活監視・イベント収集・通知。初号案件が独自実装するより早く提供する | 初号案件は独自実装の逃げ道を持つ競争状態。設定・FW管理はフェーズ2 |
| D4 | 中央集約がデフォルト、配布は例外(将来方針) | メタデータ・イベントは保護策(アクセス制御・仮名化等)で守り中央基盤に集約。基盤ごとの配布は契約上の明文要求がある案件のみ | 「出せない」の多くは社内リスク推定であり、契約実文言の確認が先。ただし初号案件は例外側からスタートする |
| D5 | 最小構成スタート | シングルノード等、公式サポートされる最小構成から開始。Kubernetesクラスタ自前運用構成は専任運用体制を持てるまで採用しない | チームはk8s/DB本番運用経験が薄い(開発はできるが運用は薄い) |
| D6 | アプライアンス型提供 | 標準機1〜2機種を基盤チームが選定・仕様を握る。調達は案件予算。「この箱を置けば動く」形態。顧客VM提供要求は例外とし、指定OSイメージそのまま稼働を条件とする | 全拠点同一構成の強制がフリート管理成立の前提。拠点ごとの個性を許すと運用崩壊 |
| D7 | エージェントの責務分割 | カメラ死活監視(ONVIF/ping)+イベント受け口(プラグイン形式)を基盤標準部品としてアプライアンスに同梱。AI製品ごとのイベント変換定義と案件固有ロジックは案件側。リポジトリは最初から分離 | 「案件固有ロジックは案件側」原則の最初の適用例。2件目以降の前例になる |
| D8 | IdPを初期同梱 | Keycloak等をアプライアンス標準構成に含め、拠点内完結のSSO(案件アプリ⇔基盤UI)を提供。人の認証(SSO/OIDC)とマシン認証(APIトークン)は分離 | 対象ユーザーは基盤チーム・案件チーム・顧客担当者・外部アプリの全て。後からの全拠点SSO後付けは移行コストが大きい |
| D9 | 通信原則: 拠点発アウトバウンドのみ | 中央/自社側から拠点への着信接続は設けない。保守はVPN経由(顧客許可済み見込み) | FW審査の通りやすさ、攻撃面の最小化 |
| D10 | 構成の版管理 | 拠点構成の差分(例: IdPあり/なし)は「構成バージョン」として管理し、拠点ごとの手作業カスタムを禁止する | 全拠点同一構成の原則と要件差分の両立 |
| D11 | サポートモデル(案・要合意) | 基盤チームは平日日中の二次対応+月次パッチ提供に限定。夜間休日の一次対応は案件側の責務とし、一次対応手順書・予備機交換手順・メーカーオンサイト保守で移譲する。復旧目標は「翌営業日」を基本線、超える要求は案件負担 | 薄い体制で24時間対応は持続不可能。※初号案件チームとの合意が必要 |
| D12 | 費用モデル | 初期2年は共通予算による投資フェーズと明確に宣言。3件目以降に軽い案件課金(実費+運用分担、独自構築より明確に安い水準)へ移行 | 無償恒久提供は要求膨張を招き、高額課金は案件離反を招く |
| D13 | VMS連携は抽象レイヤー経由 *(2026-07-06追加)* | 顧客既存VMSに合わせる前提で、基盤は製品中立インターフェース(カメラ解決・再生URL・クリップエクスポート等の最小5操作)と準拠テストキット、ONVIF Profile G汎用アダプターを提供。製品固有アダプターは案件側が実装。抽象レイヤーは映像バイト列を中継しない | 案件ごとにVMSが異なるため製品差はアダプターに閉じ込める。映像を基盤に通すとD1(帯域・コスト)が崩れる |
| D14 | Data Lakeは拠点内設置 *(2026-07-06追加)* | 映像クリップ+メタデータの長期保存用Data Lakeを各拠点閉域内に設置。保存対象はAlarm昇格分の自動保存+手動指示分のみ(全量録画はVMSの責務)。AI学習用データの拠点外持ち出しは顧客と別途合意(オフライン搬送含む) | 閉域制約(D9)と整合。全量保存はコスト目標を破壊するため「価値のある切片」に限定 |
| D15 | 映像解析AIは基盤アプライアンスと別筐体 *(2026-07-06追加)* | 映像解析AI(検知イベント生成)はCumulocity Edgeが動くアプライアンスとは別の拠点内サーバー(GPU筐体等)で動作。AI側は基盤の標準ペイロード規約(検知イベント形式、modelVersion必須)にのみ従う | GPU要件・AI製品の多様性を標準サイトサーバー像(D6)に持ち込まない。基盤停止中も検知・録画が継続する分離構成になる |
| D16 | 標準エージェントのランタイムはthin-edge.io *(2026-07-06追加)* | 基盤標準エージェント(D7)はthin-edge.io(Cumulocity開発元がメンテナンスするOSS, Apache 2.0)をランタイム基盤とし、その上にONVIF死活監視プラグインとイベント受け口プラグインを実装する。カメラはCumulocityに直接接続せず、thin-edgeゲートウェイのchild deviceとして代理登録する | Cumulocity接続の配管(証明書認証・mapper・オフラインバッファ)を自作せず開発元メンテのOSSに委ねる。閉域ライセンス問題なし。設定管理・SW更新の標準オペレーションがフェーズ2(設定・FW管理)の足場になる。※ONVIF等カメラ側ロジックは自前実装のまま。child device数百台規模はPoC確認要 |

## 3. 製品選定の必須基準(確定分)

1. **運用の軽さ**(最上位): シングルノード〜小規模構成で本番運用が公式サポートされること
2. **閉域・オフライン環境でのライセンス運用が現実的**であること(認証・更新がオンライン前提でないこと)
3. **小さくデプロイできる**こと(アプライアンス化に適すること)
4. **OIDC対応**(IdPを差し込めること)
5. **スケールパス**: 数千台への構成変更対応が可能なこと(1000台オーダー想定)
6. **エージェント開発の足場**: SDK・thin-edge.io・MQTTライブラリ等の充実度
7. **3年TCO**: 10〜50インスタンス前提での総所有コスト

## 4. 前提となった事実・状況

- Cumulocityは有力候補だが契約前。比較検討の余地あり
- 顧客の一部に「データを外に出せない」要件。ただし現時点では**契約実文言ではなく社内のリスク推定**
- 初期ユースケースは監視カメラ(駅・空港・工場・作業場所等の複数拠点)
- 自社データセンター/サーバールーム資産あり(ラック・電源・NW・仮想化基盤)
- チームは開発はできるが運用(k8s・DB本番運用)は薄い
- プロジェクト数: 初年度1、2年目までに3。カメラは多くて1000台オーダー
- 初号案件は基盤と並行開発。基盤利用の意向はあるが**独自実装の逃げ道あり**(コミット未確定)
- 保守経路: データは閉域のまま、保守用VPNは顧客が許可する見込み
- 費用: 初期は共通予算(確定)

## 6. 最重要リスクと直近アクション

**リスク第1位は技術ではなく、初号案件チームとの関係。** 基盤の提供が遅れるほど案件は独自実装に傾き、「利用者ゼロの基盤が完成する」が最悪シナリオ。次点は「案件の納期に追われて案件固有機能を基盤に作り込み、2件目で使えない基盤になる」。

### 直近アクション

1. **アンカーテナント合意の締結**(初号案件チームと)
   - 初号案件は基盤の第一利用者であり、基盤の初期スコープは初号案件の要件を最優先で決める
   - 案件固有ロジックは基盤ではなく案件側アプリに置く(D2/D7の境界線)
   - 基盤チームは最小3点セット(D3)を案件の統合テスト開始日までに提供する
   - サポート責務分担(D11)の合意
   - 独自構築想定の概算見積り(H6)の提供を受ける
2. **製品比較PoC+3年TCO見積り**(期限付き)
   - ThingsBoard CE vs Cumulocity Edge を閉域アプライアンス構成で比較
   - 拠点数10〜50前提のTCO。Cumulocityには実トポロジーを開示して見積り取得
3. **フリート管理の骨格構築**
   - 構成コード化(Ansible等)+メタ監視を1拠点目から適用。後付けは不可能
4. **契約実文言の確認**(H5)と**顧客拠点のネットワーク実態調査**

## 7. 用語・略語

- **アンカーテナント**: 基盤の第一利用者となる案件。基盤の初期要件を定義する役割
- **フリート管理**: 散在する多数の基盤インスタンスを一括で構成管理・更新・監視する仕組み
- **メタ監視**: 「基盤を監視する基盤」。各拠点インスタンスの死活・バージョンを自社側に集約
- **アプライアンス型提供**: 標準ハードウェア+構成済みソフト一式をセットで提供する形態
- **VMS**: Video Management System(映像管理システム)。Milestone、Genetec、Nx Witness等
- **thin-edge.io**: Cumulocity開発元がメンテナンスするエッジデバイス向けOSSフレームワーク(Apache 2.0)。ローカルMQTTバス・Cumulocityへのmapper・証明書認証・child device管理・設定/SW更新オペレーションを提供する
